# ADR-0004: PostgreSQL و MongoDB معًا

## الحالة

مقبول

## السياق

"النسخة" منصة للإنتاج السينمائي تخزّن نوعين متمايزين من البيانات:

1. **بيانات هيكلية محددة المخطط**: مستخدمون، جلسات، مشاريع، مشاهد، شخصيات، لقطات، رموز تحديث JWT، سجلات ZK Auth. هذه البيانات ذات علاقات واضحة ومتعددة الروابط (projects → scenes → shots).
2. **بيانات وثائق غير هيكلية**: نتائج تحليل AI، مخرجات "Seven Stations Pipeline"، بيانات وصفية للسكريبتات، استجابات نماذج LLM المتغيرة الشكل.

قاعدة بيانات واحدة ستُكلّف إما مرونة (إذا اخترنا SQL الصارم) أو موثوقية العلاقات (إذا اخترنا NoSQL).

## القرار

اعتمدنا **PostgreSQL** (عبر Neon Serverless) للبيانات الهيكلية العلائقية، و**MongoDB** للبيانات الوثائقية غير المنتظمة. البنيتان تعملان في آن واحد بتهيئة منفصلة.

الدليل من الكود:
- `apps/backend/package.json` يتضمن `"@neondatabase/serverless": "^1.0.2"` و`"mongodb": "^7.0.0"` كتبعيتين منفصلتين.
- `apps/backend/src/db/schema.ts` يعرّف جداول `users`, `projects`, `scenes`, `characters`, `shots`, `refreshTokens`, `sessions` بعلاقات مفاتيح خارجية صريحة.
- `apps/backend/scripts/test-mongodb-connection.ts` يوضح أن MongoDB له سكريبت اختبار اتصال مستقل.
- `apps/backend/src/server.ts` يستدعي `closeDatabase()` عند الإيقاف، وهو يُغلق كلا الاتصالين.

## البدائل المدروسة

| البديل | المزايا | العيوب |
|---|---|---|
| **PostgreSQL وحده مع JSONB** | بنية موحدة، لا overhead تزامن، استعلامات SQL كاملة على البيانات الوثائقية | قيود على البيانات غير المنتظمة، أداء أبطأ على وثائق ضخمة غير هيكلية |
| **MongoDB وحده** | مرونة كاملة، مناسب لبيانات AI | لا مفاتيح خارجية، صعوبة ضمان التوافق المرجعي بين المستخدمين والمشاريع والمشاهد |
| **PostgreSQL + Redis JSON** | Redis للبيانات المؤقتة سريعة الوصول | Redis ليس قاعدة بيانات دائمة، يستهلك ذاكرة عشوائية RAM لكل شيء |
| **CockroachDB** | SQL موزع، توافق PostgreSQL | تكلفة أعلى، latency أكثر من Neon، لا ميزة واضحة للمشروع الحالي |

## النتائج

### إيجابية
- **تطابق النموذج مع البيانات**: جداول SQL للبيانات ذات العلاقات (مشروع ↔ مشاهد ↔ شخصيات)، MongoDB للوثائق ذات الشكل المتغير (مخرجات نماذج AI).
- **استعلامات SQL للتقارير**: الاستعلامات المعقدة على المشاريع والمشاهد والمستخدمين تستفيد من JOIN وFOREIGN KEY وINDEX الخاصة بـ PostgreSQL.
- **مرونة MongoDB للـ AI**: استجابات LLM تأتي بتنسيقات متغيرة حسب النموذج والمهمة؛ MongoDB يستوعبها دون migration.
- **Neon Serverless**: يوفر PostgreSQL بتوسع تلقائي (auto-scaling) مناسب للبيئات الإنتاجية بدون إدارة خادوم.

### سلبية
- **تعقيد التزامن**: عمليات تمس كلتا قاعدتي البيانات (مثل حذف مشروع وبياناته التحليلية) تتطلب معالجة يدوية لضمان الاتساق؛ لا توجد معاملات (transactions) تمتد عبر PostgreSQL وMongoDB.
- **إضافة اتصالين**: كل بداية تشغيل للخادوم يُنشئ اتصالًا بكلتا قاعدتي البيانات، مما يرفع زمن الاستعداد (startup time) قليلًا.
- **اختلاف واجهات الاستعلام**: الفريق يجب أن يُتقن Drizzle ORM لـ PostgreSQL وMongoDB driver مباشرةً في نفس الوقت.
