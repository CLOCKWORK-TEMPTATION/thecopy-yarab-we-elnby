---

## 1) النطاق

- الاختبار محصور في تطبيق BREAKAPP فقط على الرابط أعلاه.
- ممنوع توسيع الاختبار إلى أي صفحة أو تطبيق آخر داخل المنصة.
- ممنوع الخلط مع تطبيقات أخرى مثل BUDGET أو cinematography-studio أو development أو styleIST.
- أي انتقال إلى تسجيل الدخول أو لوحة التحكم بعد التوجيه يُسجل كحالة اختبار، لكن المحور يبقى على البوابة نفسها.
- أي خلل خارج التطبيق لا يُسجل إلا إذا منع اختبار التطبيق نفسه بشكل مباشر.
- ممنوع تعديل كود التطبيق أو إصلاح أي عطل.

---

## 2) قواعد حاكمة غير قابلة للتفاوض

- لا تعتمد على قراءة الكود كدليل نجاح أو فشل، الدليل الوحيد المقبول هو ما يُلتقط فعليًا من المتصفح.
- لا تعتبر مجرد وجود زر أو عنصر دليل نجاح؛ يجب التفاعل معه والتحقق من نتيجة فعلية.
- لا تعتبر تحميل الصفحة وحده نجاحًا.
- لا تستخدم تقارير سابقة كدليل نجاح أو فشل.
- لا تُسقط حالة اختبار لأن الميزة غير واضحة؛ ابحث عن النظير الوظيفي.
- وثّق كل فشل بدليل مرئي أو سجل Console أو Network أو HAR.
- استخدم حالات: ناجح، فاشل، محجوب، غير موجود فقط.
- ممنوع استخدام "ناجح جزئيًا".
- ممنوع تشغيل Playwright أو Cypress أو أي إطار اختبار آلي.
- ممنوع استخدام Docker أو اقتراحه.

## 3) إعداد التنفيذ

- متصفح سطح مكتب حديث.
- دقة 1440 × 900، ثم اختبار بصري عند 768 × 1024.
- DevTools مفتوح: Console، Network، Storage، Cookies، Performance.
- نافذة نظيفة لكل اختبار توجيه وجلسة.
- استخدام Offline لاختبار انقطاع الشبكة.
- استخدام Axe DevTools أو Lighthouse للوصولية.
- التقاط لقطات شاشة قبل وبعد كل خطوة، وتسجيل سلسلة Redirects كاملة.
- تسجيل HAR لكل فشل، خاصة لرصد سلاسل التوجيه.

## 6) مصفوفة الاختبارات الخاصة بـ BREAKAPP

### BRKAPP-01 — تحميل التطبيق
- الخطوات: افتح الرابط في نافذة نظيفة.
- المتوقع: تحميل البوابة وعرض حالة معالجة قصيرة قبل التوجيه.
- الدليل المطلوب: لقطة + Console + Network.

### BRKAPP-02 — سلوك المستخدم غير المسجل
- الخطوات: افتح الرابط دون أي كوكيز جلسة.
- المتوقع: توجيه إلى صفحة تسجيل الدخول بشكل واضح ومتوقع.
- الدليل المطلوب: لقطة + سلسلة Redirects + Network.

### BRKAPP-03 — إعادة التوجيه إلى login أو dashboard إن حدثت
- الخطوات: راقب رمز الحالة وعنوان الوجهة بعد التوجيه.
- المتوقع: 302 أو 307 نظيفة إلى وجهة منطقية.
- الدليل المطلوب: HAR.

### BRKAPP-04 — حفظ الرابط المقصود بعد تسجيل الدخول إن ظهر
- الخطوات: حاول الوصول إلى مسار محدد، توجّه إلى تسجيل الدخول، ثم ادخل.
- المتوقع: العودة إلى المسار المقصود بعد الدخول.
- الدليل المطلوب: لقطات + Network.

### BRKAPP-05 — عدم وجود redirect loop
- الخطوات: راقب طلبات Network أثناء التوجيه.
- المتوقع: عدم وجود حلقة 3xx متكررة.
- الدليل المطلوب: HAR.

### BRKAPP-06 — زر الرجوع في المتصفح بعد التوجيه
- الخطوات: بعد التوجيه، اضغط زر الرجوع في المتصفح.
- المتوقع: سلوك متوقع ومفهوم دون كسر.
- الدليل المطلوب: لقطة + Network.

### BRKAPP-07 — مسح الكوكيز ثم إعادة التحميل
- الخطوات: امسح كوكيز النطاق، أعد تحميل الصفحة.
- المتوقع: العودة إلى سلوك المستخدم غير المسجل.
- الدليل المطلوب: لقطة + Cookies + Network.

### BRKAPP-08 — اختبار دور مستخدم إن توفرت حسابات
- الخطوات: ادخل بحسابات بأدوار مختلفة (إن توفرت).
- المتوقع: توجيه مختلف وفق الدور.
- الدليل المطلوب: لقطات لكل دور.

### BRKAPP-09 — مسار غير صحيح تابع للتطبيق
- الخطوات: افتح BREAKAPP/path-not-exist.
- المتوقع: رسالة 404 أو توجيه آمن دون كشف بنية داخلية.
- الدليل المطلوب: لقطة + Network.

### BRKAPP-10 — قياس سرعة التوجيه
- الخطوات: قِس الزمن بين دخول الرابط والوصول إلى الوجهة.
- المتوقع: زمن منخفض دون تأخير غير مبرر.
- الدليل المطلوب: تقرير Performance.

### BRKAPP-11 — عدم عرض بيانات جلسة أو رموز
- الخطوات: راقب URL و Network و Console.
- المتوقع: عدم ظهور توكنات أو معرفات جلسة في URL أو سجلات.
- الدليل المطلوب: HAR + Console.

### BRKAPP-12 — تسجيل حالات الحجب بسبب غياب بيانات الدخول
- الخطوات: إذا تطلبت حالة اختبار حسابًا غير متاح، وثّقها كمحجوبة.
- المتوقع: تسجيل ما جُرّب قبل الحجب.
- الدليل المطلوب: ملاحظات + لقطات.

## 8) معايير قبول تنفيذ الاختبار

- لا يُقبل التقرير النهائي إلا إذا غطى كل الحالات في الأقسام 5 و 6 و 7.
- لا يُقبل تسجيل النجاح بدون دليل.
- لا يُقبل تسجيل الفشل بدون خطوات إعادة إنتاج.
- لا تُقبل الحالة المحجوبة إلا بعد توضيح ما جُرّب وسبب التعذر.
- لا يُقبل خلط هذا التطبيق مع أي تطبيق آخر داخل التقرير.
- ممنوع منعًا تامًا أن يحتوي هذا التقرير على أي محتوى يخص BUDGET أو cinematography-studio أو development أو styleIST أو أي تطبيق غير BREAKAPP.
- لا يُقبل الحكم Production-Ready إلا إذا لم توجد إخفاقات حرجة أو مرتفعة غير محلولة.
- لا يُقبل أي فحص يكتفي بقراءة الكود أو نسخ نتائج تقرير قديم.
- يجب الالتزام بصيغة التقرير في القسم 4 حرفيًا.
