## 1) الحكم التنفيذي على التطبيق

**فاشل تشغيليًا للمستخدم غير المسجَّل** — استوديو التحليل الدرامي السباعي يعرض واجهة احترافية، يقبل النص، ويُطلق طلب POST صحيح إلى الخادم (200 OK) عند الضغط على `ابدأ التحليل`. غير أن طلبَي
`/api/public/analysis/seven-stations/stream/{id}`
و
`/api/public/analysis/seven-stations/{id}/snapshot`
**يعودان كلاهما 403 Forbidden**، مما يجعل خط أنابيب التحليل عالقًا عند 5% بلا تقدّم وبلا رسالة خطأ. التقرير القديم ادّعى نجاح كل المحطات السبع — هذا الادعاء **مدحوض ميدانيًا** لأن المحطات لا تظهر أصلًا للمستخدم غير المسجَّل.

التطبيق يكشف هنا تناقضًا في التفويض على مستوى الـ Backend: نقطة `/start` تقبل المجهول، ونقاط القراءة (stream/snapshot) ترفضه. النتيجة: التحليل يبدأ على الخادم ويُهدر قدرة حسابية، والمستخدم لا يرى نتيجته.

## 3) الأعطال المؤكدة

### حرج (Critical)

**1. تناقض تفويض في خط أنابيب التحليل**
- `POST /api/public/analysis/seven-stations/start` يقبل المجهول ويُرجِع 200 + UUID.
- `GET /api/public/analysis/seven-stations/stream/{id}` يُرجِع **403**.
- `GET /api/public/analysis/seven-stations/{id}/snapshot` يُرجِع **403**.
- النتيجة: الخادم يُهدر طاقة في تشغيل تحليل لا يستطيع المستخدم رؤية نتيجته.

**2. الفشل الصامت في الواجهة عند 403**
- لا يُعرض أي toast أو alert أو رسالة خطأ.
- شريط التقدم عالق عند `قيد التحليل ٠٥%` بلا حركة.
- المستخدم يبقى في انتظار وهمي بلا تفسير.

### مرتفع (High)

**3. FCP بطيء جدًا**
- `first-contentful-paint = 7400ms` — أكثر من ضعف العتبة المقبولة (2500ms عند Lighthouse Good).
- يضاعف معدل الارتداد (Bounce Rate).

**4. عدم استرجاع حالة التحليل بعد reload**
- URL يحتوي `?analysis={uuid}` لكن الواجهة لا تستخدمه لاستعادة الحالة.
- textarea عاد فارغًا، المحطات لا تُعاد عرضها، لا رسالة `جلسة سابقة موجودة`.

**5. غياب رسائل خطأ على مستوى الواجهة**
- 403 من backend لا تتحول لرسالة مفهومة للمستخدم.
- تجربة مستخدم متدنية: يبدو أن التطبيق معطّل بلا تفسير.

### متوسط (Medium)

**6. عدم وجود رفع ملفات (PDF/TXT)**
- `inputs[type=file].length = 0` — السيناريو يجب لصقه يدويًا، لا يمكن رفع ملف Final Draft أو PDF.
- ميزة طبيعية مفقودة لتطبيق تحليل سيناريوهات.

## 6) أخطر المخاطر التشغيلية

**1. Backend يُهدر طاقة على تحليلات بلا متلقٍ**
- كل ضغطة `ابدأ التحليل` من مستخدم غير مسجَّل تُنشئ تحليلًا فعلًا (200 من `/start`)، ثم تُحجب نتيجته.
- إذا كان التحليل يستهلك LLM tokens (أسعار GPT-4 مثلًا)، فهذا تسريب مالي مباشر.

**2. تجربة المستخدم الكارثية للزائر**
- زائر يدخل التطبيق، يكتب نصًا، يضغط `ابدأ التحليل`، يرى شريط 5% عالقًا للأبد.
- لا يعرف أنه يحتاج تسجيل دخول، لا يعرف أن هناك خطأ، لا يعرف أن النتيجة لن تظهر.

**3. URL يحتفظ بـ analysis ID بلا فائدة**
- بعد reload يصبح URL=`?analysis={uuid}` معلقًا. لا الـ ID يُستخدم لاسترجاع، ولا يُمحى من URL، ولا يُطلب تسجيل دخول لاسترجاعه.

## 7) خطوات إعادة إنتاج كل فشل

### إعادة إنتاج الفشل الحرج #1 — 403 على stream/snapshot

1. افتح
`https://www.thecopy.app/analysis`
في تبويب نظيف غير مسجَّل.
2. ألصق أي نص عربي (>50 حرف) في textarea بعنوان `ألصق النص الدرامي هنا لبدء التحليل`.
3. افتح DevTools → Network filter `analysis`.
4. انقر على زر `ابدأ التحليل`.
5. **النتيجة الفعلية:**
- `POST /api/public/analysis/seven-stations/start` → 200 OK + JSON يحوي UUID
- `GET /api/public/analysis/seven-stations/stream/{uuid}` → **403**
- `GET /api/public/analysis/seven-stations/{uuid}/snapshot` → **403**

### إعادة إنتاج الفشل الحرج #2 — الفشل الصامت

1. كرر الخطوات 1-4 أعلاه.
2. **النتيجة الفعلية:** بعد 35+ ثانية:
- شريط التقدم عالق عند `قيد التحليل ٠٥%`.
- لا يظهر toast، لا alert، لا رسالة `يجب تسجيل الدخول`.
- المحطات السبع لا تظهر بطاقاتها.

### إعادة إنتاج الفشل المرتفع #3 — FCP 7400ms

1. افتح DevTools → Performance.
2. ابدأ تسجيل ثم navigate إلى `/analysis`.
3. **النتيجة الفعلية:** `first-contentful-paint` ≥ 7400ms.

### إعادة إنتاج الفشل المرتفع #4 — عدم استعادة الحالة

1. أكمل خطوات إنتاج الفشل #1.
2. لاحظ URL تحول إلى `?analysis={uuid}`.
3. اضغط F5.
4. **النتيجة الفعلية:** بعد التحميل (~7s)، textarea فارغ، المحطات لا تُعاد عرضها، لا رسالة استرجاع.

- إجمالي 75 char، محجوبة عن العرض في Chrome MCP.

## 10) فحص الأمان الأساسي

| الفحص | النتيجة | التفصيل |
|---|---|---|
| XSS (script/img onerror) | آمن | `xssExecutedFlag: false`، React escape سليم |
| SQL Injection (مدخل JSON body) | غير قابل للاختبار من الواجهة | الإدخال يذهب POST كـ JSON، الـ backend هو من يتعامل معه |
| إدخال طويل (112K) | يقبل بلا حد | لا حد أعلى، لكن لا انهيار |
| كشف stack trace | لم يُلاحَظ | 403 لا يكشف stack — جيد |
| تسريب JWT في URL | لا | URL يحوي `?analysis={uuid}` فقط |
| كشف مفاتيح API في DOM/Console | لم يُلتقط (Console محجوب) | — |
| Sentry monitoring | مفعَّل | `POST /monitoring?o=4511284205453312` |
| نقطة `/start` تقبل المجهول | **مخاطرة** | ينفّذ تحليلًا للمجهول دون أي حماية ضد abuse — لا rate limiting واضح، يمكن أن يستنزف موارد LLM |

## 11) توصيات الإصلاح مرتبة حسب الأولوية

### حرجة

**T1.** **توحيد سياسة التفويض على المسارات الثلاثة** (`/start`, `/stream/{id}`, `/{id}/snapshot`):
- إما أن تكون كلها مفتوحة للمجهول (مع rate limiting صارم).
- أو كلها محمية بـ auth، مع رفع نقطة `/start` نفسها لـ 401 قبل تشغيل التحليل على الخادم.
- الخيار الثاني هو الموصى به لتجنب تسريب موارد LLM.

**T2.** **معالج خطأ موحَّد للأكواد 401/403 على مستوى الواجهة:**
- toast واضح: `يلزم تسجيل دخول لاستعراض نتائج التحليل` + زر `سجِّل دخولًا`.
- إيقاف شريط التقدم وعرض حالة `محجوب` بدلًا من `قيد التحليل`.

**T3.** **معالج timeout** على شريط التقدم: إذا لم يتحرك خلال 30 ثانية، يعرض `انتهت المهلة، أعد المحاولة`.

### مرتفعة

**T4.** **خفض FCP من 7400ms إلى أقل من 2500ms:**
- مراجعة الـ critical rendering path.
- تأجيل تحميل المكونات غير الأساسية (lazy loading للمحطات السبع).
- streaming SSR مناسب.

**T5.** **استعادة الحالة عبر `?analysis={uuid}` في URL:**
- عند تحميل الصفحة مع وجود معلَمة `analysis` في URL، يجب أن تُحاول الواجهة جلب snapshot.
- إن نجح: استعرض النتائج.
- إن فشل بـ 403: اعرض `يلزم تسجيل دخول لاستعراض هذا التحليل`.
- إن فشل بـ 404: اعرض `هذا التحليل غير موجود` ونظِّف URL.

**T6.** **إضافة autosave لـ textarea** قبل بدء التحليل:
- `localStorage.setItem('analysis.draft', ...)` كل 5 ثوانٍ + on blur.
- استرجاع المسودة عند فتح الصفحة.

### متوسطة

**T7.** **إضافة دعم رفع ملفات** (PDF, TXT, Final Draft) في الواجهة:
- يقلل الاحتكاك للمستخدم الذي لديه سيناريو طويل في ملف.

**T8.** **إضافة rate limiting على `/start`** لمنع abuse من المجهول.

**T9.** **عرض تفاصيل الخطة المتاحة بدون حساب:**
- مثلًا: محطة واحدة مجانية، باقي المحطات بعد التسجيل.
- يتجنب فشل صامت كامل.

### منخفضة

**T10.** **توحيد localStorage namespace** بين تطبيقات `the copy` لتجنب تسرب الحالة.

**T11.** **إضافة Lighthouse و Axe DevTools** كمتطلب في CI.
