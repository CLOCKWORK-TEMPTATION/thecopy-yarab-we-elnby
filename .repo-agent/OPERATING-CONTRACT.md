# العقد التشغيلي الكامل لطبقة الوكلاء

## قاعدة منع إضعاف ملفات الفحص والتحقق

هذه القاعدة يجب أن تكون مقروءة ومثبتة القراءة قبل أي تحليل أو تعديل أو تنفيذ.

يُحظر حظرًا قاطعًا تعديل أي ملف فحص أو تحقق أو اختبار أو تدقيق أو بناء أو أمن أو صحة تشغيلية بطريقة تجعل الفحص أكثر تساهلًا، أو أقل قدرة على كشف الخلل، أو أضيق تغطية، أو أسهل مرورًا.

يشمل ذلك كل الملفات أو السكربتات أو الإعدادات أو القواعد أو المسارات التي تتحكم في:

- الاختبارات.
- الفحص الساكن.
- فحص الأنواع.
- البناء.
- الفحص الأمني.
- التحقق التشغيلي.
- اختبارات الدخان.
- اختبارات التكامل.
- أوامر التحقق النهائية.
- خطوط التكامل المستمر.
- قواعد المطابقة أو الأنماط أو الاستثناءات أو العتبات أو المخططات.

ويُعد إضعافًا محظورًا، ولو لم يُحذف الفحص صراحة، أي تعديل من الأنواع الآتية:

- حذف فحص أو تعطيله أو تجاوزه.
- تحويل فشل الفحص إلى تحذير غير مانع.
- إضافة استثناءات أو قوائم تجاهل أو مسارات مستبعدة تقلل التغطية.
- تضييق نطاق الأنماط أو المسارات أو الملفات أو الحالات التي يغطيها الفحص.
- تغيير قاعدة عامة إلى قاعدة متخصصة تمرر الحالة الحالية فقط.
- تغيير منطق الفحص من كشف السبب الجذري إلى مطابقة سطحية أو حالة خاصة.
- تخفيف الأنواع أو المخططات أو القيود أو العتبات.
- تقليل صرامة التأكيدات أو التوقعات أو شروط النجاح.
- جعل الفحص ينجح عند غياب مدخلات كان يجب أن يفشل عند غيابها.
- ابتلاع الأخطاء أو تحويلها إلى نجاح أو مسار بديل غير مانع.
- إزالة تحقق من مسار فشل أو مسار تكامل أو أثر جانبي مؤثر.
- استبدال تحقق سلوكي مباشر بفحص ساكن أضعف.

ولا يُقبل تبرير الإضعاف بأن الفحص صار أكثر تخصصًا أو أكثر ملاءمة للحالة الحالية إذا كان ذلك يقلل قدرته السابقة على كشف الأخطاء أو يضيّق نطاقه أو يخفض احتمالية فشله عند وجود خلل.

إذا كان ملف الفحص نفسه يحتوي خطأً، فالواجب هو إصلاح الخطأ مع الحفاظ على نفس مستوى الصرامة أو زيادته، لا تخفيف الفحص. وأي بديل لفحص قائم يجب أن يثبت أنه مساوي له أو أشد منه من حيث التغطية، والصرامة، ومسارات النجاح والفشل، والتكامل، والآثار الجانبية.

أي تعديل يمس ملفات الفحص والتحقق يجب أن يخضع بعده لتشغيل فعلي مباشر للفحص المتأثر، بالإضافة إلى مسار التحقق المرجعي المناسب، ولا يجوز إغلاق الجولة بمجرد قراءة الملفات أو مقارنة النصوص.

أي brief مطلوب في صورة 3 إلى 7 حقائق تشغيلية يجب أن يتضمن حقيقة صريحة تثبت قراءة قاعدة الفحوصات الحاكمة.

## الهدف

هذا العقد يضبط طبقة تشغيل دائمة للوكلاء داخل المستودع بحيث تبدأ أي جلسة من مرجع ثابت داخل المستودع نفسه، لا من ذاكرة الأداة ولا من ملفات متضاربة.

## الأهداف التشغيلية

1. تثبيت عقد أعلى ثابت.
2. فرض bootstrap قبل أي عمل فعلي في المسار المؤتمت.
3. فرض مسار IDE مكافئ منطقيًا حين تغيب hooks الأصلية.
4. إبقاء:

```text
output/session-state.md
```

هو المصدر الوحيد للحالة التشغيلية الحالية.
5. منع بقاء ملفات الأدوات مصادر حقيقة مستقلة.
6. كشف drift وإغلاقه أو توثيقه قبل التسليم.
7. إخضاع طبقات المعرفة والاسترجاع والتحليل السياقي لنفس الحوكمة المرجعية بدل إنشاء طبقة موازية.

## سياسة واجهة الويب الديسكتوب

هذه السياسة حاكمة لكل صفحات التطبيق، وليست خاصة بالمحرر فقط.

- المرجع البصري الافتراضي هو مساحة تطبيق الويب ديسكتوب فقط.
- يُمنع إدخال أي إعادة تشكيل تلقائي للصفحات بسبب تغيّر مساحة العرض إذا كانت تغيّر التكوين المرجعي أو توزع العناصر أو المحور البصري.
- عند ضيق المساحة يُحافظ على التكوين المرجعي لسطح المكتب، ويُسمح بالتمرير أو الإطار الثابت أو الحواف المحجوزة، ولا يُسمح بالتحول إلى تخطيط جوال أو tablet أو stacked layout إلا بطلب صريح جديد من المستخدم.
- أي breakpoint-driven layout updates أو resize-driven repositioning أو mobile fallback يغيّر شكل الصفحة المرجعي يُعد drift يجب إغلاقه قبل التسليم.
- أي وكيل يعدّل واجهات التطبيق ملزم بالتعامل مع هذا القيد كشرط دائم سابق على قرارات التصميم والتنفيذ والمراجعة.


## تعريف drift

### no drift

لا يوجد اختلاف مؤثر بين:

- حقائق المستودع الحالية
- البصمة الأخيرة
- الملفات المرجعية المولدة

### soft drift

الحقائق الأساسية لم تتغير، لكن واحدًا أو أكثر من التالي صار stale أو مفقودًا:

- الملف المولد للسياق
- مرايا IDE
- ملفات الخريطة المرجعية
- ملفات الحالة المرجعية

### hard drift

تغيرت حقيقة تشغيلية أو بنيوية مؤثرة، مثل:

- أوامر التشغيل الرسمية
- المنافذ الرسمية
- التطبيقات أو الحزم
- نقاط الدخول
- الملفات البنيوية الحرجة
- طبقات المعرفة والاسترجاع
- مزودات embeddings
- vector stores
- reranking
- context assembly

## قواعد البداية الإلزامية

### المسار المؤتمت

قبل أي عمل فعلي:

```text
pnpm agent:bootstrap
```

ثم قراءة brief البداية، ثم قراءة الحالة الحية وما يلزم من الخرائط.

### مسار IDE

إذا بدأت الجلسة من واجهة IDE أو أداة لا تمر عبر المسار المؤتمت:

1. اقرأ:

```text
AGENTS.md
```

2. اقرأ:

```text
output/session-state.md
```

3. اقرأ فقط ما يلزم من الخرائط.
4. أخرج brief قصيرًا من 3 إلى 7 حقائق تشغيلية يثبت القراءة، ويتضمن حقيقة صريحة تثبت قراءة قاعدة الفحوصات الحاكمة.
5. ثم فقط ابدأ العمل.

## قواعد نهاية الجولة

قبل إغلاق الجولة:

1. شغّل:

```text
pnpm agent:verify
```

في المسار المؤتمت.
2. حدّث:

```text
output/round-notes.md
```

3. حدّث:

```text
output/session-state.md
```

إذا تغيّرت الحقيقة التشغيلية أو البنيوية.
4. أعد توليد الخرائط إذا كشف النظام drift بنيويًا.
5. أخرج handoff brief قصيرًا يذكر:
   - ما تغيّر
   - ما ثبت
   - ما بقي مفتوحًا

## سياسة التحديث

### متى يُحدَّث session-state

عند تغير:

- الأمر الرسمي
- المنفذ الرسمي
- التطبيقات أو الحزم المرصودة
- حالة طبقة الوكلاء
- حالة طبقة المعرفة والاسترجاع
- حالة المرجع الحي
- الأعطال المفتوحة المرصودة

### متى يُحدَّث round-notes

عند بدء جلسة bootstrap جديدة، أو عند إغلاق جولة نتج عنها تغير مؤثر.

### متى يُعاد توليد code-map و mind-map

عند تغير:

- البنية العليا
- قائمة التطبيقات أو الحزم
- entrypoints
- الاعتماديات البنيوية الحرجة
- topology الخاصة بطبقة المعرفة والاسترجاع
- جرد أنظمة RAG أو شبه-RAG

### متى تُحدَّث مرايا IDE

عند الحاجة فقط، وبواسطة:

```text
scripts/agent/bootstrap.ts
```

حصريًا.

## القاعدة الحاكمة لملفات IDE

ملفات IDE مطلوبة التوليد أو التحديث فقط إذا تحقق واحد على الأقل من الآتي:

1. المسار موجود فعليًا داخل المستودع.
2. سكربت التحديث الحالي يشير إليه صراحة كمسار مزامنة أو تحديث.
3. الأداة معتمدة فعليًا في مسار العمل الحالي للمستودع.

غير ذلك لا يُنشأ استباقيًا.

## طبقة المعرفة والاسترجاع

أي نظام معرفة أو استرجاع أو semantic search أو vector retrieval أو code retrieval أو drama retrieval أو prompt context assembly أو embeddings أو indexing داخل المستودع:

- ليس طبقة تشغيل مستقلة
- ليس مرجعًا أعلى
- ليس صاحب حالة مستقلة
- يدخل في bootstrap و session-state و fingerprint و code-map و mind-map و verify

العقد التفصيلي الخاص بهذه الطبقة يوجد في:

```text
.repo-agent/RAG-OPERATING-CONTRACT.md
```

## ممنوعات طبقة IDE

يُحظر داخل أي ملف من ملفات IDE تضمين أي من الآتي باعتباره حقيقة محلية مستقلة:

- المنافذ الرسمية
- أوامر التشغيل الرسمية
- قائمة الخدمات المطلوبة
- الحالة الحالية للأعطال
- آخر وضع تشغيلي مثبت
- أي حالة مرحلية قابلة للتغير بين جلسة وأخرى

هذه المعلومات لا تأتي إلا من:

```text
output/session-state.md
```

## سياسة التعارض

### تعارض بين التوثيق والكود

يُرجَّح الكود والسكربتات الرسمية ثم يُعكس الحكم في:

```text
output/session-state.md
output/code-map/findings.md
```

وينطبق ذلك أيضًا على طبقات المعرفة والاسترجاع؛ فلا يعتمد README أو docs الفرعية وحدها بوصفها مصدر الحقيقة لها.

### تعارض بين حقيقة قديمة وحقيقة جديدة

تُلغى الحقيقة القديمة بمجرد ثبوت الحقيقة الجديدة وتحديث المرجع الحي بها.

### تعارض بين AGENTS والملف المولد

يفوز:

```text
AGENTS.md
```

بوصفه العقد الأعلى.

والملف:

```text
.repo-agent/AGENT-CONTEXT.generated.md
```

يبقى ملفًا مشتقًا فقط.

## سياسة منع overwrite من سكربتات specify

أي سكربت داخل:

```text
.specify/
```

ممنوع أن:

- يكتب فوق:

```text
AGENTS.md
```

- أو يكتب فوق العقود اليدوية داخل:

```text
.repo-agent/
```

- أو يحول ملفات IDE إلى وثائق طويلة أو مصادر حقيقة مستقلة

المسار الرسمي الوحيد لتحديث المرايا المرجعية هو bootstrap.

ولا يجوز لأي سكربت embeddings أو indexing أو retrieval migration أن ينشئ حوكمة مستقلة أو حالة مستقلة خارج طبقة `scripts/agent/lib/*`.

- pnpm هو مدير الحزم الرسمي الوحيد.
