# بروتوكول البداية

## قاعدة منع إضعاف ملفات الفحص والتحقق

هذه القاعدة يجب أن تكون مقروءة ومثبتة القراءة قبل أي تحليل أو تعديل أو تنفيذ.

يُحظر حظرًا قاطعًا تعديل أي ملف فحص أو تحقق أو اختبار أو تدقيق أو بناء أو أمن أو صحة تشغيلية بطريقة تجعل الفحص أكثر تساهلًا، أو أقل قدرة على كشف الخلل، أو أضيق تغطية، أو أسهل مرورًا.

يشمل ذلك الاختبارات، والفحص الساكن، وفحص الأنواع، والبناء، والفحص الأمني، والتحقق التشغيلي، واختبارات الدخان، واختبارات التكامل، وأوامر التحقق النهائية، وخطوط التكامل المستمر، وقواعد المطابقة أو الأنماط أو الاستثناءات أو العتبات أو المخططات.

ويُعد إضعافًا محظورًا حذف فحص أو تعطيله أو تجاوزه، أو تحويل فشله إلى تحذير غير مانع، أو إضافة استثناءات تقلل التغطية، أو تضييق نطاقه، أو تخفيف الأنواع أو المخططات أو القيود أو العتبات، أو إزالة تحقق من مسار فشل أو تكامل أو أثر جانبي مؤثر، أو استبدال تحقق سلوكي مباشر بفحص ساكن أضعف.

إذا كان ملف الفحص نفسه يحتوي خطأً، فالواجب إصلاح الخطأ مع الحفاظ على نفس مستوى الصرامة أو زيادته. وأي بديل لفحص قائم يجب أن يثبت أنه مساوي له أو أشد منه من حيث التغطية، والصرامة، ومسارات النجاح والفشل، والتكامل، والآثار الجانبية.

أي تعديل يمس ملفات الفحص والتحقق يجب أن يخضع بعده لتشغيل فعلي مباشر للفحص المتأثر، بالإضافة إلى مسار التحقق المرجعي المناسب، ولا يجوز إغلاق الجولة بمجرد قراءة الملفات أو مقارنة النصوص.

أي brief مطلوب في صورة 3 إلى 7 حقائق تشغيلية يجب أن يتضمن حقيقة صريحة تثبت قراءة قاعدة الفحوصات الحاكمة.

## قاعدة البداية

قبل أي تحليل أو تعديل أو تنفيذ فعلي:

```text
pnpm agent:bootstrap
```

ويجب أن يبدأ هذا الأمر بالحارس الآلي الذي يقرأ:

```text
.repo-agent/TOOL-GUARD-CONTRACT.json
```

الحارس الآلي قبل البداية مسؤول عن تحديث ذاكرة الكود الحية عند التقادم أو النقص، وعن منع البداية إذا لم يمكن الوصول إلى ذاكرة محدثة وقابلة للتحقق.

## التسلسل الإلزامي

1. شغّل:

```text
pnpm agent:bootstrap
```

2. اقرأ brief البداية الناتج.
3. اقرأ:

```text
output/session-state.md
```

4. اقرأ فقط الأجزاء اللازمة من:

```text
output/code-map/*
output/mind-map/*
```

5. بعد ذلك فقط ابدأ المهمة الفعلية.

إذا كانت المهمة تمس طبقة المعرفة والاسترجاع أو embeddings أو indexing أو context assembly:

6. اقرأ ما يلزم من:

```text
.repo-agent/RAG-OPERATING-CONTRACT.md
output/code-map/rag-systems.md
output/code-map/rag-entrypoints.md
output/mind-map/rag-topology.mmd
```

## إذا فشل bootstrap

لا تبدأ المهمة.

يجب أولًا إغلاق سبب الفشل أو توثيقه ثم إعادة التشغيل.

## إذا كشف bootstrap drift

- إذا كان drift قابلًا للتسوية الآلية فدع bootstrap يكمله.
- إذا بقي drift بعده فلا يبدأ العمل قبل تسويته أو توثيقه صراحة.

## التشغيل الآلي مع أي خطوة

كل أمر رسمي يمثّل بداية خطوة تطوير أو بناء أو اختبار أو فحص أو بحث في الذاكرة يجب أن يملك خطافًا سابقًا يشغل الحارس الآلي.

التحقق النهائي يجب أن يثبت وجود هذه الخطافات، لا أن يفترضها.

## إذا غيّرت المهمة حقيقة تشغيلية أو بنيوية

قبل التسليم:

- حدّث:

```text
output/session-state.md
```

- حدّث:

```text
output/round-notes.md
```

- وأعد توليد الخرائط إذا صار drift البنيوي قائمًا.

وينطبق ذلك أيضًا على أي تغير في:

- أنظمة RAG
- مزودي embeddings
- vector stores
- reranking
- query routing
- context assembly

## مسار IDE عند غياب hooks الأصلية

إذا كانت الأداة لا تنفذ الأوامر تلقائيًا:

1. اقرأ:

```text
AGENTS.md
```

2. اقرأ:

```text
output/session-state.md
```

3. اقرأ فقط ما يلزم من الخرائط.
4. أخرج brief قصيرًا من 3 إلى 7 حقائق تشغيلية، ويتضمن حقيقة صريحة تثبت قراءة قاعدة الفحوصات الحاكمة.
5. ثم فقط ابدأ التنفيذ.

هذا التسلسل هو بداية الجلسة الرسمية في مسار IDE.

ولا تُستثنى طبقة المعرفة والاسترجاع من هذا التسلسل.
