# بروتوكول إغلاق الجولة

## قاعدة منع إضعاف ملفات الفحص والتحقق

هذه القاعدة يجب أن تكون مقروءة ومثبتة القراءة قبل أي تحليل أو تعديل أو تنفيذ.

يُحظر حظرًا قاطعًا تعديل أي ملف فحص أو تحقق أو اختبار أو تدقيق أو بناء أو أمن أو صحة تشغيلية بطريقة تجعل الفحص أكثر تساهلًا، أو أقل قدرة على كشف الخلل، أو أضيق تغطية، أو أسهل مرورًا.

يشمل ذلك الاختبارات، والفحص الساكن، وفحص الأنواع، والبناء، والفحص الأمني، والتحقق التشغيلي، واختبارات الدخان، واختبارات التكامل، وأوامر التحقق النهائية، وخطوط التكامل المستمر، وقواعد المطابقة أو الأنماط أو الاستثناءات أو العتبات أو المخططات.

ويُعد إضعافًا محظورًا حذف فحص أو تعطيله أو تجاوزه، أو تحويل فشله إلى تحذير غير مانع، أو إضافة استثناءات تقلل التغطية، أو تضييق نطاقه، أو تخفيف الأنواع أو المخططات أو القيود أو العتبات، أو إزالة تحقق من مسار فشل أو تكامل أو أثر جانبي مؤثر، أو استبدال تحقق سلوكي مباشر بفحص ساكن أضعف.

إذا كان ملف الفحص نفسه يحتوي خطأً، فالواجب إصلاح الخطأ مع الحفاظ على نفس مستوى الصرامة أو زيادته. وأي بديل لفحص قائم يجب أن يثبت أنه مساوي له أو أشد منه من حيث التغطية، والصرامة، ومسارات النجاح والفشل، والتكامل، والآثار الجانبية.

أي تعديل يمس ملفات الفحص والتحقق يجب أن يخضع بعده لتشغيل فعلي مباشر للفحص المتأثر، بالإضافة إلى مسار التحقق المرجعي المناسب، ولا يجوز إغلاق الجولة بمجرد قراءة الملفات أو مقارنة النصوص.

أي brief مطلوب في صورة 3 إلى 7 حقائق تشغيلية يجب أن يتضمن حقيقة صريحة تثبت قراءة قاعدة الفحوصات الحاكمة.

## القاعدة العامة

لا يُعتبر العمل منتهيًا إلا إذا أُغلقت الجولة مرجعيًا، لا بمجرد توقف التعديل.

## متى يجب تشغيل verify

شغّل:

```text
pnpm agent:verify
```

قبل التسليم في كل جولة غيّرت ملفات الطبقة المرجعية أو طبقة الوكلاء أو البنية.

ويجب أن يثبت هذا الأمر وجود العقد الآلي والخطافات المانعة وأوامر الحارس، وأن يثبت أن ذاكرة الكود الحية ليست قديمة مقارنة ببصمات ملفات الكود الحرجة.

## بوابة إغلاق الذاكرة الدائمة

قبل أي تسليم يخص طبقة الوكلاء أو الذاكرة الدائمة أو الحكم على الجاهزية، شغّل:

```text
pnpm agent:persistent-memory:session:close
```

ولا يمر الإغلاق إذا بقي أي دور بلا:

```text
turn_context_status
query_hash
selected_intent
retrieval_event_id
audit_event_id
memory_context
```

ويجب تشغيل:

```text
pnpm agent:persistent-memory:turn:verify
```

لإثبات أن سياق السؤال الحي المولد الحالي قابل للاستهلاك.

## تحقق الحارس الآلي

لا يكفي وجود الأوامر نصيًا.

إغلاق الجولة يجب أن يشمل تشغيلًا فعليًا لأمر حارس واحد على الأقل، وتشغيلًا فعليًا لمسار تحقق شامل يعتمد على الحارس السابق للأمر.

## متى يجب تحديث session-state

يجب تحديث:

```text
output/session-state.md
```

إذا تغيّرت:

- حقيقة تشغيلية
- حقيقة بنيوية
- حالة طبقة الوكلاء
- حالة طبقة المعرفة والاسترجاع
- حالة المرجع الحي

## متى يجب إضافة جولة في round-notes

يجب إضافة أو إلحاق جولة عند:

- بدء جلسة bootstrap جديدة
- إنهاء جولة نتج عنها تغير حقيقي
- إغلاق drift كان مؤثرًا

## متى يجب إعادة توليد الخرائط

عند تغير:

- التطبيقات أو الحزم
- entrypoints
- أوامر التشغيل الرسمية
- طبقة الوكلاء العليا
- أنظمة المعرفة والاسترجاع
- rag entrypoints
- rag topology

## handoff brief الإلزامي

يجب أن يحدد handoff brief النهائي:

1. ما الذي تغيّر
2. ما الذي تم التحقق منه مباشرة
3. ما الذي بقي مفتوحًا
4. هل استلزم الأمر تحديث:

```text
output/session-state.md
output/round-notes.md
output/code-map/*
output/mind-map/*
```

## مسار IDE

في مسار IDE لا يُعتبر العمل مغلقًا إلا إذا:

1. تم تحديث:

```text
output/round-notes.md
```

2. تم تحديث:

```text
output/session-state.md
```

عند تغير الحقيقة التشغيلية أو البنيوية. 3. تم إخراج handoff brief قصير يذكر:

- ما الذي تغيّر
- ما الذي ثبت
- ما الذي بقي مفتوحًا

## قاعدة عدم التسليم الخاطئ

أي تغيير حقيقي في التشغيل أو البنية أو طبقة الوكلاء دون تحديث المرجع الحي يجعل التسليم غير صحيح.

وينطبق الحكم نفسه على أي تغيير حقيقي في طبقة المعرفة والاسترجاع أو embeddings أو indexing أو retrieval.
